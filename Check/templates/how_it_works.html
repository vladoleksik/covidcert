Here, you can find more about your - and everyone else's - cerificate. On this page, we briefly present how it is designed, structured, signed and verified - while doing away with centralised databases.

The design of the certificate

The main idea of this certificate is to avoid using centralised databases. If every vaccinated person has their details stored in a large table (which is primarily used to generate and verify "certificates"), not only would you have to give the read credentials (i.e. username and password or API key) to everyone at every border, event entry, restaurant, school etc., but you would also have to give the full read-write permissions to every Ministry, Hospital test point in order to be able to generate all the certificates with a decent level of flexibility. This way, it is only a matter of time until some of the credentials get leaked out and, once unauthorised access to the database is granted, the damage is hard to even estimate. What is more, you would need Internet access to be able to check the validity of any claim. This is the pre-digital equivalent of having a centralised table of everyone in existence and their current status. Insecure, sluggish, inconvenient.

The main idea behind this certificate system is the equivalent of hand-signed diplomas, passports and other documents. There is no need to actually compare any certificate to a register entry, just to see if the one who signed it is someone you trust (a competent authority). More explicitly, it uses a set of national databases only as a backup, while the verification procedure is done on-site, by perforing a set of mathematical checks. This way, we determine if the certificate corresponds to a set of rules that can only be followed by the respective issuer - no forgeries!

When you enter your data on a certificate generation site (of your national Ministry of Health or equivalent authority), the institution can check if your data is correct, then write a short note: "I, the Utopian Ministry of Health, state that this person is ... and therefore ok. This note is valid from today for one year." Then, it performs a mathematical function on the entire message (every piece of text is, after all, a number) and attaches the result to the message. This function depends on a secret number only known by the respective institution and, therefore, can only be applied to the original message by that institution. Also, the inverse function is made public and can be used to un-jumble any message and verifiy its provenience.

The verifying side uses that inverse function, who it knows it comes from your issuer, applying it to the attached signature and, if it un-jumbles to the exact same message, it certifies the authenticity of the message.

How a message is actually signed

The only mathematically possible way to provide a decentralised certification system is to come up with an asymmetric function system. In a nutshell, a pair of two functions: jumble(x) and unjumble(x). Essentially, it is imperative that x is equal to unjumble(jumble(x)). Also, the jumble function has to be secret, thus incredibly hard to determine, even if the unjumble function is known and public. Even more, it is neessary to find a method to, quickly and reliably, find a unique set of such functions whenever necessary. One of the first known examples is called RSA (Rivest-Shamir-Adleman) and is in use even today in most cryptographic systems in the world.

The key is, surprisingly, in prime numbers! Take two prime numbers, p and q (for example, 3 and 17, but the numbers used in reality are about 1024 digits long each!) and multiply them: n=p*q. After that, take a standard number e (usually 65537, but now, 7). Now we want to find d, the number with (xe*d) mod n = x, for any x, and equal to the modular inverse of e mod phi(n). The latter is a mathematical function which, for the case of RSA, evaluates to phi(n)=(p-1)*(q-1). d can be calculated by taking any k for which (1+k*(phi(n))/e is an integer. That integer is d. For our example, n=51, phi(n)=2*16=32. Thus we find k=5, for which d=23.

So, jumble is: jumble(x)=xd mod n or x23 mod 51

And unjumble is: unjumble(x)=xe mod n or x7 mod 51

One can observe that, as it is very difficult to compute p and q knowing only n, it is unfeasible to find the signing function by just knowing the verifying function.

Although RSA is not used in this certificate, another similar algorithm is used: the Elliptic Curve Digital Signature Algorithm (ECDSA). The main idea remains the same: take your message, compute its hash (a shorter, obscure string that is easier to encrypt, but (ideally) uniquely identifies a message). Now, pass the hash through the jumbling function and attach the result to your message. Make sure your unjumbling algorithm is known by everyone and you can publish your message that can now be undeniably linked to you and just you.
